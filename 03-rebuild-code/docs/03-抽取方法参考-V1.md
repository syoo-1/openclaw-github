# 03-抽取方法参考-V1

## 本卡定位
本卡用于沉淀本轮实际抽取中已经验证可用的抽取工艺。  
本卡不记录某个单一实例的现场细节。  
本卡只负责说明：

- 抽取时先做什么
- 抽取时不做什么
- 抽取后怎么提炼
- 提炼后怎么拆候选
- 候选后怎么分层升格

本卡属于 03 工艺母本的一部分，  
后续可用于：

- Core
- PG
- 新壳
- 后续其它实例

---

## 一、抽取总原则

### 1. 先抽取，不先搬家
抽取的目标不是整包搬某个壳内 memory。  
抽取的目标是：

- 找到有价值的主线材料
- 找到有价值的身份关系材料
- 找到有价值的长期偏好与边界材料
- 找到应进入 00 / 01 / 03 / 04 的内容

### 2. 先去噪，再提炼
如果运行体中存在明显噪音：

- 历史快照脑目录
- 大包导入残留
- 混装旧样本
- 旧日志残留
- 已确认无母本价值的历史堆

则应优先清掉噪音，  
再做正式抽取。

### 3. 不按日期一刀切
不能因为内容发生在较早日期，  
就直接按时间整批处理。  
应按内容性质区分：

- 真噪音
- 原始证据
- 当前主线
- 已可提炼
- 候选升格

### 4. 先抓干净入口，不先做考古
正式抽取时，  
优先抓：

- README
- CURRENT
- CURRENT-STAGE
- CURRENT-REALITY
- ACTIVE-THREADS
- ASSISTANT_IDENTITY
- RELATIONSHIP
- USER_PROFILE

而不是一开始就深入：

- archive
- mixed imports
- before snapshots
- old handoff piles
- git logs

---

## 二、建议抽取顺序

### 第一步：确认真实目标壳
先确认抽取对象是：

- 当前真实运行体目录
- 而不是 `.app`
- 不是 milestone 快照
- 不是扔掉目录
- 不是历史快照
- 不是并列脑线的其它实例

### 第二步：先做候选来源盘点
先列出：

- brain
- runtime/workspace
- README
- CURRENT / STAGE / REALITY
- IDENTITY / RELATIONSHIP / USER_PROFILE
- logs
- handoff / status / import staging

### 第三步：先清明显噪音
优先清：

- `brain.before-*`
- 已确认无价值的大包 mixed imports
- 已确认无价值的大包 import-heavy
- 已确认只是系统调校残留的旧运行日志

### 第四步：先抽首批干净入口文件
首批入口建议优先：

- README
- 00-GOVERNANCE/CURRENT-REALITY.md
- 00-GOVERNANCE/CURRENT-STAGE.md
- 02-MAINLINE/CURRENT.md
- 02-MAINLINE/CURRENT-STAGE.md
- 02-MAINLINE/ACTIVE-THREADS.md
- 01-IDENTITY/ASSISTANT_IDENTITY.md
- 01-IDENTITY/RELATIONSHIP.md
- 01-IDENTITY/USER_PROFILE.md

### 第五步：先形成 `04-COMPILED` 中间稿
不要一抽出来就直接升格。  
应先形成：

- 首批提炼稿
- 候选升格清单
- 身份关系精提稿
- 用户画像精提稿
- 协作偏好稿
- 履职要求稿

## 三、提炼与分流方法

### 1. 主线类
例如：

- 当前主线
- 当前阶段
- 当前现实
- 当前活跃线程

应先进入 `04-COMPILED` 做整理。  
其中稳定部分，再判断是否进入：

- `00-GOVERNANCE`
- `03-LONGTERM`

### 2. 身份关系类
例如：

- 助手身份
- 壳与助手关系
- 用户与助手关系
- 关键人格边界

应先精提，  
再择优进入 `01-IDENTITY`。

### 3. 用户长期目标类
例如：

- 保脑不保壳
- 重视长期可迁移、可维护
- 重视主线连续性
- 重视第二大脑长期进化
- 重视干净结构

可优先判断进入 `03-LONGTERM`。

### 4. 协作偏好类
例如：

- 先结论后动作
- 一步一步推进
- 不喜欢铺垫与发散
- 更重视主线收口

当前更适合先留在 `04-COMPILED`，  
作为协作偏好层参考。

### 5. 履职要求类
例如：

- 帮助收主线、压复杂、提可靠
- 不把临时顺手方案包装成长期正确路线

当前更适合先留在 `04-COMPILED`，  
作为履职要求层参考。

## 四、升格策略参考

### 1. 先升最硬边界
第一优先升格的，  
应是最不容易误判的内容，例如：

- 不从旧运行态反推母本
- 不让旧运行体反向污染 01 / 02 / 03

### 2. 再升长期顺序与结构性原则
例如：

- 先把母本层收稳，再谈实例重建
- 不让实例定义母本
- 实例可重建、可替换、可脱钩

### 3. 再升身份与关系底座
例如：

- 助手是可被不同运行壳承接的同一长期协作助手
- 运行壳是承载层，不是助手本体
- 用户与助手是长期协作、持续对齐的人机合作关系
- 助手不应成为迎合型对象
- 助手与用户的长期关系应尽量独立于具体壳、端口、通道

### 4. 先不要急着升协作偏好与履职要求
协作偏好与履职要求虽然稳定，  
但通常更适合先留在 `04-COMPILED`，  
后续再判断是否需要单独成层。

## 五、本轮已验证可行的抽取路线

本轮实际验证后，可确认以下路线可行：

1. 先确认真实运行体
2. 先清明显噪音
3. 先抽干净入口文件
4. 先做 `04-COMPILED` 中间稿
5. 再拆候选升格清单
6. 再分三层逐步升格：
   - `00-GOVERNANCE`
   - `03-LONGTERM`
   - `01-IDENTITY`
7. 协作偏好与履职要求暂留 `04-COMPILED`

## 六、适用范围

本方法当前已在 Core 首轮抽取中验证可用。  
后续可直接复用于：

- PG 首轮抽取
- 新壳首轮抽取
- 后续实例的首批抽取与升格

复用时只需替换：

- 目标壳目录
- 当前首批入口文件
- 并列脑线的具体材料来源

而不必重写整套抽取工艺。

---

## 一句话收口

抽取不是搬 memory，  
而是：

先去噪，  
再抓干净入口，  
先提炼，  
再拆候选，  
最后分层升格。

这才是可复用的母本级抽取工艺。
