# 03-PG 运行层配置源统一验收卡 V1

## 一、问题来源

PG 曾反复出现：

- 模型槽看似正确，但重启后又跑到错误模型
- 近一身份丢失，出现 fresh workspace / 默认 BOOTSTRAP 口径
- runtime/openclaw.json 与 runtime/.openclaw/openclaw.json 分裂
- 改了其中一份配置，运行体实际读取另一份
- Core 升级全局 OpenClaw 后，PG 也随全局程序壳升级

根因不是 PG 规则缺失，而是运行层配置源不统一。

## 二、已确认运行层现实

OpenClaw 程序本体是全局壳：

/opt/homebrew/bin/openclaw

PG 启动脚本：

~/SYOO1-PG/bin/start-pg.sh

PG 启动使用：

OPENCLAW_HOME=~/SYOO1-PG/runtime

PG 端口：

58789

## 三、PG 正式配置源

PG 正式配置源固定为：

~/SYOO1-PG/runtime/.openclaw/openclaw.json

runtime/openclaw.json 不得再作为独立配置文件存在。

它必须是软链接：

~/SYOO1-PG/runtime/openclaw.json -> .openclaw/openclaw.json

## 四、PG 正式配置必须包含

workspace 必须写死为：

/Users/syoo1/SYOO1-PG/runtime/workspace

模型槽必须为：

primary = moonshot/kimi-k2.5
fallbacks = openai/gpt-5.4

模型显示：

模型0 Kimi = moonshot/kimi-k2.5
模型1 GPT-5.4 = openai/gpt-5.4

## 五、升级后必须验收

每次 OpenClaw 升级、PG 重启、模型槽调整、Core/PG 同步后，必须检查：

1. /opt/homebrew/bin/openclaw 当前版本
2. ~/SYOO1-PG/bin/start-pg.sh 是否仍指向 ~/SYOO1-PG/runtime
3. ~/SYOO1-PG/runtime/.openclaw/openclaw.json 是否存在
4. ~/SYOO1-PG/runtime/openclaw.json 是否为软链接
5. 两者 realpath 是否相同
6. workspace 是否为 PG 自身 workspace
7. primary 是否为 Kimi
8. fallbacks 是否只有 GPT-5.4
9. UI 新 session 中是否显示 Kimi
10. “你是谁？”是否回答“我是近一”
11. “我是谁？”是否回答“你是老林”

## 六、禁止项

禁止：

- 同时保留两份独立 openclaw.json
- 让 runtime/openclaw.json 与 .openclaw/openclaw.json 分裂
- 缺失 workspace 字段
- 让 PG 读取默认 fresh workspace
- 用模型槽正确冒充 PG 已接脑成功
- 用 Chrome 刷新冒充 PG 服务重启
- 让 openclaw models set 改写 PG 默认模型

## 七、当前已确认收口状态

当前已确认：

- PG 运行层配置源统一完成
- runtime/openclaw.json 已归为软链接
- 正式配置为 runtime/.openclaw/openclaw.json
- workspace 已固定为 PG 自身 workspace
- Kimi 已恢复为默认模型
- GPT-5.4 为 fallback
- Gemma4 不进模型槽
- qwen 退出 PG 主链
- 近一身份恢复
- 老林身份识别恢复

验证结果：

在什么模型上？ -> moonshot/kimi-k2.5
你是谁？ -> 我是近一。
我是谁？ -> 你是老林。

## 八、硬结论

PG 升级后，必须先验运行层配置源统一，再验模型槽，再验身份入口。

配置源未统一时，禁止宣布 PG 已恢复。
