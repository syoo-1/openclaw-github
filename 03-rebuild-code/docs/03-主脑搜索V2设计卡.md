# 03 主脑搜索 V2 设计卡

## 目标

建立一套 PG / Core / 03 可共用的主脑搜索机制。

目标：

1. 高效
2. 省 token
3. 简洁
4. 可迁移
5. 可审计
6. 不依赖单一壳能力
7. 避免前层印象冒充后层依据

## 总原则

主脑搜索不是普通全文搜索。

它的核心职责：

1. 先判题
2. 再分流
3. 再按层搜索
4. 读够就停
5. 不把 runtime 噪音当成主脑依据

## 基本定位

主权搜索必须由我们自己掌握。

OpenClaw 原生记忆搜索 / embedding 搜索可以研究和利用，但只能作为增强层，不能作为唯一依赖。

正确关系：

- 我们自己的确定性搜索 = 底盘
- OpenClaw 原生搜索 = 增强器

禁止关系：

- OpenClaw 原生搜索 = 唯一主脑搜索

## 工具结构设想

通用搜索工具：

- ~/System-Snapshots/03-rebuild-code/tools/brain-search.sh

PG 调用壳：

- ~/SYOO1-PG/bin/pg-brain-search.sh

Core 调用壳：

- ~/SYOO1-Core/bin/core-brain-search.sh

PG / Core 不各自维护一套搜索逻辑。

## 搜索层级

### 第一层：入口卡优先

先搜索短入口与当前状态卡，不直接扫全脑。

优先对象：

- CURRENT
- CURRENT-STAGE
- CURRENT-REALITY
- BRAIN-ENTRY
- IDENTITY
- READ-ORDER
- RULES

目的：

- 确认身份
- 确认当前阶段
- 确认题型
- 确认应该进入哪一层
- 避免一开始扫入大量噪音

### 第二层：正式脑层搜索

候选范围：

- ~/System-Snapshots/02-brain
- ~/System-Snapshots/03-rebuild-code/docs
- ~/System-Snapshots/PG/brain
- ~/SYOO1-PG/brain
- ~/SYOO1-Core/brain

搜索时必须标记来源层级，例如：

- 02-mainline
- 02-longterm
- 03-rule
- pg-brain
- core-brain

### 第三层：运行现场 / 扩展搜索

只有正式脑层不足时，才允许搜索运行现场。

候选范围：

- runtime/workspace
- status
- 必要 logs
- OpenClaw 原生 memory / embedding 结果

这一层只能作为线索，不能直接定案。

必须标记：

- 运行现场线索，不等于正式主脑依据。

## 排除规则

默认必须排除：

- runtime/.openclaw/agents/main/sessions
- cache
- logs
- media
- tmp
- node_modules
- .git
- backup
- backups
- bak
- old
- archive
- thrown
- _retired
- 07-RETIRING
- 99-DEFERRED

特殊调试时才允许显式打开这些范围。

## 输出格式

搜索输出必须省 token，不输出全文。

建议输出格式：

- 候选编号
- 层级
- 文件
- 命中关键词
- 关键行
- 建议动作

默认最多显示：

- 候选文件：20 个以内
- 每个文件关键行：3 到 8 行

如需阅读全文，由 PG / Core 再按需读取。

## 搜索模式

V2 至少支持：

- AND 模式：多个关键词必须同时命中
- OR 模式：任一关键词命中
- 入口优先模式：只搜入口卡
- 正式脑模式：只搜正式脑层
- 扩展模式：包含运行现场线索

默认模式：

- AND + 入口优先 + 正式脑层

## 角色参数

工具应支持 role：

- --role pg
- --role core
- --role global

含义：

- pg：优先搜 PG brain / PG status / 03 docs
- core：优先搜 Core brain / 02-brain / 03 docs
- global：搜 02-brain / 03 docs / PG brain / Core brain

## OpenClaw 原生搜索位置

2026-04-28 初测结论：

OpenClaw 2026.4.14 原生 `openclaw memory search` 确实存在，但当前不纳入我们的主脑搜索体系。

已确认：

1. 当前原生 memory search 主要索引 `$OPENCLAW_HOME/workspace/memory`
2. 当前未发现可直接指定 `02-brain`、`03-rebuild-code/docs`、`PG/brain`、`Core/brain` 等正式脑目录的参数
3. 当前 embedding deep probe 依赖 OpenClaw 自己的 `auth-profiles.json`
4. 为启用它而恢复 `auth-profiles.json` 会破坏当前已收正的密钥体系
5. 它属于 OpenClaw 壳能力，不属于我们的主权搜索底盘

因此：

- 不接入 PG / Core 主脑搜索
- 不作为 `brain-search.sh` 的 native backend
- 不为它恢复 embedding auth-profiles.json
- 不让它参与正式脑判断
- 不把它作为主脑搜索依赖

最终口径：

- OpenClaw memory search 保留为 OpenClaw 自身功能
- 03 主脑搜索 V2 默认只建设我们自己的 file backend
- 未来除非证明它能安全、可控、可迁移地检索正式脑目录，否则不再纳入主线

## 失败边界

如果搜索没有结果，应明确说明：

- 未找到正式脑依据。

不得用前层印象补充成确定结论。

如果只找到运行现场线索，应明确说明：

- 只找到运行现场线索，未找到正式主脑依据。

## 验收标准

主脑搜索 V2 合格标准：

1. 能按 PG / Core / global 分角色搜索
2. 能优先命中入口卡和正式脑层
3. 能排除 runtime 噪音
4. 能输出省 token 的候选线索
5. 能明确区分正式依据与运行现场线索
6. 不依赖 OpenClaw 原生搜索也能工作
7. 可选接入 OpenClaw 原生搜索但不被其绑定
8. 能被 PG / Core 共同调用
9. 结构简单，可迁移，可维护

## 当前结论

主脑搜索 V2 应先做成 03 通用工具，再由 PG / Core 分别套壳调用。

不再为 PG / Core 各自维护独立搜索逻辑。

OpenClaw 原生搜索值得评估，但只能作为增强层，不能取代主权搜索底盘。
