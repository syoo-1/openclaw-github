# 03-gateway-token时序卡

## 定位
本卡用于定义：

**gateway token 的获取、注入与验证，必须纳入正式壳适配工艺。**

它不是后层补充动作，
也不是“跑通后再想起来处理”的附属事项。

## 当前已验证现实
在新壳实验中，已出现以下现实：

- 即使壳表面已启动
- 页面可能已能打开
- 服务看起来也可能已在运行
- 但如果 gateway token 没在合适时机获取、放对、带入
- 后面仍然无法顺利接上助理

因此：

**token 没处理对时序时，表面跑通不等于真实接通。**

## 正式规则

### 规则 1
新壳首次适配时，必须显式确认 gateway token 的获取方式。

### 规则 2
gateway token 不得拖到“壳表面跑通之后再临时处理”。

### 规则 3
至少在首配完成前后这一段工艺内，
必须明确：
- token 从哪里获取
- 注入到哪里
- 当前控制面是否实际使用了该 token

### 规则 4
不允许把“页面能打开 / 服务能启动 / Dashboard 可访问”误判为助理已可正常承接。

### 规则 5
若 token 未正确获取或未正确注入，
本轮不视为正式通过。

## 最小验收项
至少应确认：

1. 已取得当前实例应使用的 gateway token
2. token 已被放入正确位置
3. 控制面使用的是当前实例 token，而不是旧残留
4. 助理可基于该 token 顺利接上，不只是页面可开

## 当前结论
gateway token 不是附属细节，
而是：

**壳适配是否真正接通助理的硬门槛之一。**

## 一句话收口
token 放错位置或放错时机，
就会出现：
**壳像跑通了，但助理其实没真正接上。**
