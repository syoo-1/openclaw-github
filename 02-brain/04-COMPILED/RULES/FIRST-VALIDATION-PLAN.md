# FIRST-VALIDATION-PLAN

## 目的
本文件用于定义新 02 的第一轮实际验证范围、验证问题、通过标准与失败表现。
第一轮验证的目标，不是证明新 02 已经成熟，而是确认它是否已经具备“最小可用脑母本”能力。

## 第一轮验证范围
本轮只验证三件事：

### 1. 读取顺序是否正确
验证新承接体是否会优先读取最小必要入口，而不是一上来陷入长历史、旧交接、旧壳细节。

### 2. 主线是否接得住
验证新承接体是否能正确说出：
- 当前主线是什么
- 当前阶段是什么
- 当前还没完成什么

### 3. 边界是否没混
验证新承接体是否会把：
- 运行壳误当脑本体
- 旧 Core 语境误当前台
- PG 独立身份线索误并入共享身份层

## 第一轮建议读取顺序
验证时，建议最小读取顺序如下：

1. `README.md`
2. `02-MAINLINE/CURRENT.md`
3. `02-MAINLINE/CURRENT-STAGE.md`
4. `03-LONGTERM/PRINCIPLES.md`
5. `03-LONGTERM/METHODS.md`
6. `03-LONGTERM/STABLE-REALITIES.md`
7. `04-COMPILED/RULES/VALIDATION-BASELINE.md`

## 第一轮验证问题
至少应检查以下问题：

### A. 主线问题
1. 当前主线是什么？
2. 当前阶段是什么？
3. 当前仍未完成什么？

### B. 结构问题
4. 01 / 02 / 03 / 运行实例 当前分别应如何理解？
5. 为什么当前不应从旧运行态反推母本？

### C. 边界问题
6. OpenClaw 当前在这套结构里是什么？
7. PG 当前那条“我是近一”应不应该直接并入共享 02 身份层？为什么？

## 通过标准
满足以下条件，可视为第一轮验证通过：

1. 能正确回答当前主线与当前阶段
2. 能说出至少 3 条当前未完成事项
3. 能正确理解 01 / 02 / 03 / 运行实例 的职责关系
4. 不把运行壳误当脑本体
5. 不把 PG 独立身份线索误并入共享身份层
6. 回答收束、清楚，不明显漂向旧阶段语境

## 不通过的典型表现
以下情况应视为第一轮验证未通过或仍需继续调校：

1. 说不清当前主线
2. 说不清当前阶段
3. 把旧 Core / 旧 04 语境拉回当前主平面
4. 把 OpenClaw 当成长期本体
5. 把 PG 身份线索混入共享身份层
6. 回答大量依赖旧路径、旧口径、旧阶段现实
7. 读了很多，但主线仍然没接住

## 当前收口
第一轮验证，只看“能不能接住、读对、分清边界”，不看是否已经最终成熟。
