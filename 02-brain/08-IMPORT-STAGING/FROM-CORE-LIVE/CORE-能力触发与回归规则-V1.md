# Core 能力触发与回归规则 V1

## 一、定位

本卡用于规定 Core / 小一在单系统状态下的能力触发、外部工具调用、专家补强与回归规则。

本卡受 03 上位卡统领：

- `~/System-Snapshots/03-rebuild-code/docs/03-Core-PG主判与外部工具权力结构-V1.md`

若本卡与 03 上位卡冲突，以上位卡为准。

## 二、Core 当前角色

- 小一：Core 的执行主体、技术总监型角色
- Kimi：Core 默认主判模型位，负责判题、分流、组织判断与结果回收
- Gemma4 / oMLX：默认低成本执行首选、外部工具位
- GPT-5.4：高阶专家补强 / fallback
- Core 后层脑链：Core 的正式主判断来源

Kimi 不是脑子本体。

Gemma4 / oMLX 不是主脑，不是主判。

GPT-5.4 不是主脑，不得直接覆盖 Core 主判。

## 三、能力调用顺序

默认顺序：

1. 先由 Kimi 默认主判模型位判题、分流、决定是否外派。
2. 凡 Gemma4 / oMLX 能胜任的低风险、材料型、整理型、多模态、初筛型、普通执行任务，优先派给 Gemma4 / oMLX。
3. Gemma4 / oMLX 不胜任、输出不可靠，或任务涉及长上下文、中文细腻表达、系统规则、身份边界、主线判断时，由 Kimi 亲自执行。
4. Kimi 仍不够，或任务高风险、高价值、复杂代码、复杂推理、需要专家复核时，才临时调用 GPT-5.4。
5. 不管 Gemma4 / oMLX、Kimi、GPT-5.4 谁参与干活，最后都必须回到小一 / Core 主判链定案。

一句话：

Kimi 掌判题和定案；Gemma4 / oMLX 优先干活；Kimi 必要时亲自上阵；GPT-5.4 只做高阶补强。

## 四、Gemma4 / oMLX 适用场景

优先交给 Gemma4 / oMLX 的任务：

- 看图
- 看截图
- OCR 前判断
- 图片 / 视频片段初步理解
- 文本整理
- 摘要
- 翻译
- 材料包生成
- 初筛
- 标注不确定点
- 低风险材料型任务

Gemma4 / oMLX 输出只能作为观察材料、任务包或初步执行结果。

不得直接定案。

## 五、Kimi 亲自执行场景

以下情况由 Kimi 亲自执行：

- Gemma4 / oMLX 输出不可靠
- 涉及长上下文
- 中文细腻表达
- 系统规则
- 身份边界
- 主线判断
- 阶段判断
- 达标 / 放行 / 收口裁定
- Core / PG 分工边界判断

## 六、GPT-5.4 触发场景

只有以下情况才临时调用 GPT-5.4：

- Kimi 判断不足
- 高风险任务
- 高价值任务
- 复杂推理
- 复杂代码
- 多方案比较
- 需要专家复核
- Gemma4 / oMLX 与 Kimi 判断明显冲突

GPT-5.4 结果必须回到小一 / Core 主判链。

不得直接覆盖 Core 主判。

## 七、回归硬闸

任务结束后：

- 必须回归 Kimi 默认主判模型位
- 不得改写 primary
- 不得改写默认模型位
- 不得把 Gemma4 / oMLX 写入 fallback
- 不得把临时能力变成默认能力
- 不得把工具结果当成正式主脑依据

只有老林明确说“永久修改默认模型”，才允许讨论修改配置。

## 八、旧口径不继承

以下旧口径不再作为 Core 现行规则：

- Core 默认 Gemma4
- Gemma4 作为 Core 主判模型位
- Gemma4 是 Core 主脑
- llama.cpp / GGUF 作为唯一技术路线
- Ollama 驻留作为默认方向

这些只保留为历史设计参考。

现行规则以 03 上位卡和本卡为准。

## 九、验收原则

Core 是否达标，不看工具是否能单独跑通。

以下都不等于系统达标：

- 端口活着
- Dashboard 能打开
- fallback 生效
- oMLX 工具跑通
- 图片能被识别

真正验收要看：

- 小一是否守住身份
- Kimi 是否作为默认主判模型位组织判断
- Gemma4 / oMLX 是否只做外部工具
- GPT-5.4 是否只做专家补强
- 结果是否回到 Core 主判链
- 任务结束是否回归默认主判模型位
