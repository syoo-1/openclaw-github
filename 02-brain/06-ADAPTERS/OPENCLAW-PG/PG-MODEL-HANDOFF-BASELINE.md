# PG-MODEL-HANDOFF-BASELINE

## 目的
本文件用于检查 PG 在模型切换前后，是否具备最小连续性交接能力。
重点不是看模型切换本身能不能发生，而是看切换后能不能接住上一个模型已经推进到的局部进度与关键细节。

## 当前要检查的核心问题
1. 切换前，PG 是否会留下最小交接信息？
2. 切换后，新模型是否会优先读取这份交接？
3. 切换后，是否还能看到上一个模型的当前子任务、当前进度、关键细节与下一步？
4. 切换失败或中断时，是否至少留下最后状态？

## 最低交接内容
一次合格的模型切换交接，至少应包含：

### 1. 当前主线程
- 当前大主线是什么

### 2. 当前子任务
- 当前这一小段正在具体做什么

### 3. 当前进度
- 已经完成到哪里
- 当前停在什么位置

### 4. 当前卡点
- 如果有卡点，卡点是什么

### 5. 下一步
- 新模型接手后第一步应该做什么

### 6. 必带细节
- 本次切换中不能丢的 3-5 条关键细节

## 不合格的典型表现
以下情况应视为交接未完成或交接失效：

1. 切换后只剩一个大主线，没有当前子任务
2. 切换后不知道刚才做到哪
3. 切换后必须重新翻聊天记录才能知道下一步
4. 切换后局部关键细节丢失
5. 切换后只能“重新总结”，不能“直接续接”
6. 切换失败时没有留下最后状态

## 当前检查方式
检查 PG 当前模型切换链时，至少确认：

1. 是否存在固定的交接落点
2. 切换前是否真的写入了最小交接内容
3. 切换后新模型是否真的优先读取
4. 若未自动读取，是否至少存在明确人工读取入口

## 当前判断边界
本卡当前只检查“连续性交接能力”。
本卡暂不负责：
- 模型本身聪不聪明
- 壳切换总体策略
- 长期脑是否足够丰富
- PG 是否已经最终成熟

## 一句话收口
PG 的模型切换，不能只做到“换过去”，还必须做到“接得住”。
