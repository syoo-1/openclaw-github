# PG-AUTO-TRIGGER-BASELINE

## 目的
本文件用于检查 PG 在突发状况下，是否具备最小自动触发与自动介入能力。
重点不是看 PG 平时能不能正常运行，而是看出现异常、中断、失败或连续不稳定时，PG 是否会自动留下状态、触发守望链或进入保护动作。

## 当前要检查的核心问题
1. PG 是否有明确的自动触发条件？
2. 发生异常时，PG 是否会自动记录当前状态？
3. 发生异常时，PG 是否会自动触发守望 / 介入 / 通知链？
4. 自动触发是否有边界，避免误触发和乱接管？

## 应检查的突发情况
至少应检查以下类型：

### 1. 模型报错
- 请求失败
- 返回异常
- 模型不可用
- provider / route 报错

### 2. 回复中断
- 中途停住
- 输出被截断
- 尚未完成即退出

### 3. 连续失败
- 同类错误连续出现
- 连续几次都无法正常完成当前任务

### 4. 切换后未接住
- 模型已切换成功
- 但新模型未接住当前子任务或局部进度

### 5. 壳仍在，但链路已失稳
- 进程未死
- 但逻辑链、交接链或响应链已异常

## 最低自动触发能力
一次合格的自动触发能力，至少应包括：

### 1. 自动留痕
异常发生时，至少自动留下：
- 当前主线程
- 当前子任务
- 当前停在哪
- 当前异常类型

### 2. 自动触发动作
异常发生后，至少自动执行其中一种：
- 触发守望链
- 触发通知
- 触发最小交接落点
- 进入保护状态

### 3. 自动触发边界
自动触发必须有清晰边界：
- 什么情况触发
- 连续几次失败后触发
- 什么情况只记录不接管
- 什么情况允许介入

## 不合格的典型表现
以下情况应视为自动触发能力未建立或未收稳：

1. 出错时没有自动留痕
2. 连续失败后没有任何自动动作
3. 切换未接住时没有补救触发
4. 只能靠人工事后发现问题
5. 自动动作存在，但触发条件不清楚
6. 自动触发过于泛滥，造成误介入或乱接管

## 当前检查方式
检查 PG 当前自动触发链时，至少确认：

1. 是否已存在自动触发脚本、规则或守望链
2. 是否已存在异常留痕位置
3. 是否已存在异常后的自动动作
4. 是否已定义触发条件与边界
5. 是否已做过至少一轮手动验通或模拟验证

## 当前判断边界
本卡当前只检查“突发状况自动触发能力”。
本卡暂不负责：
- PG 的长期脑是否足够成熟
- PG 的身份是否已最终定型
- Core 与 PG 的完整协作拓扑
- 所有异常场景是否都已完全覆盖

## 一句话收口
PG 不能只在平稳时好用；出事时也必须有人接棒，哪怕那个人先是自动触发链。
