# System-Snapshots 同步总规则卡

## 一、定位

本卡是 `System-Snapshots` 的同步总规则卡。

本卡只负责固定两类同步：

1. **本地同步**  
   运行中的正式来源对象 → `System-Snapshots` 中对应的正式母本位置

2. **GitHub 同步**  
   `System-Snapshots` → GitHub

本卡不是：

- 运行体说明
- PG 单独规则卡
- 旧 01 讨论入口
- 某一条具体同步线的临时说明

本卡生效后，今后与 `System-Snapshots` 有关的同步，都先按本卡判断，不再混线。

---

## 二、总原则

### 1）只保脑，不保壳
同步的核心目标，是保护和沉淀长期有效的正式资产，不是搬运运行现场。

### 2）宁愿少，不愿乱
同步宁可收窄，也不允许把运行体垃圾、临时文件、退役内容、日志媒体混进母本真源。

### 3）运行体不是母本
运行体只能作为正式资产的来源现场，不能被当成可整包回写的真源。

### 4）进入 System-Snapshots 之后，才有资格继续对外同步
只有已经正式进入 `System-Snapshots` 并站稳的内容，才允许继续同步到 GitHub。

### 5）自动链从严，手动权保留
自动同步必须从严收窄。  
用户本人保留必要的反向手动操作权，但反向只作为受控修复动作，不作为默认日常动作。

### 6）同步默认按“对象”理解，不按“整包现场”理解
同步的最小判断单位，应是正式对象、正式路径、正式母本位。  
不能把“整个运行目录”“整个现场目录”直接理解成同步对象。

---

## 三、GitHub 同步总规则

### 1）定义
GitHub 同步指：

`System-Snapshots → GitHub`

这是母本仓向远端仓的版本化同步，不是运行体同步。

### 2）总口径
凡是已经正式纳入 `System-Snapshots` 的内容，只要符合母本真源准入条件，就都可以随 `System-Snapshots` 整仓同步到 GitHub。

GitHub 同步认的是：

`System-Snapshots` 这个母本仓

而不是：

- 某个单独运行体
- 某个单独现场
- 只限于 02、03 的固定狭义范围

### 3）当前可被 GitHub 接纳的对象
包括但不限于：

- `02-brain`
- `03-rebuild-code`
- `PG`
- 以后新增并正式纳入 `System-Snapshots` 的其他母本级内容

### 4）前提
进入 GitHub 同步范围的前提是：  
这些内容本身已经符合 `System-Snapshots` 的正式准入条件。

不得夹带：

- 运行体
- 日志
- 媒体
- 密钥
- 临时现场
- 退役垃圾
- `.openclaw`
- `.env.local`
- 缓存
- 其他非正式母本资产

### 5）GitHub 不直接接运行体
GitHub 只能接 `System-Snapshots`。  
不允许：

- Core 直接推 GitHub
- PG 运行体直接推 GitHub
- 任何现场目录直接推 GitHub
- 把本地运行目录当成远端真源

### 6）GitHub 按整仓差异同步理解
只要 `System-Snapshots` 已经整理干净，GitHub 同步就按整个仓来理解。  
不再把 02、03、PG 等拆成互相割裂的远端思路。

### 7）手动审后推送优先
当前允许继续采用手动审后推送。  
若未来恢复自动推送，也必须建立在：

- `System-Snapshots` 长期保持干净
- 边界长期稳定
- 仓内对象长期明确

的前提下。

## 四、本地同步总规则

### 1）定义
本地同步指：

运行中的正式来源对象 → `System-Snapshots` 中对应的正式母本位置

这是运行现场向母本真源的受控沉淀，不是整包搬运，不是双向自动同步。

### 2）总口径
本地同步不只是一条固定线路，而是一组受统一原则约束的同步线。

当前有哪些正式来源对象，就可以有对应的本地同步线；以后若新增正式对象，也可以按同一原则增加新线。

### 3）本地同步的本质
本地同步不是：

- 搬家
- 整包复制
- 运行体备份
- 现场照搬

本地同步的本质是：

**把运行中的正式有效资产，经过阀门、过滤和准入判断后，沉淀进 `System-Snapshots` 对应母本位。**

### 4）默认方向
所有本地同步线，默认都按：

**来源对象 → `System-Snapshots` 对应正式母本位**

来理解。

也就是说，默认是：

- 从运行中的正式来源对象进入母本
- 不是母本自动反向灌现场
- 不是双向自动同步

### 5）自动链禁止事项
本地自动同步默认禁止：

- 自动双向同步
- 未经判断的反向覆盖
- 把母本自动回灌成运行现场
- 按整个运行目录整包同步
- 把日志、媒体、缓存、密钥等非正式资产卷入母本

### 6）用户保留反向手动权
用户本人保留对本地同步执行反向手动操作的权力。

该权力主要用于：

- 修复运行体偏移
- 回补正确正式资产
- 校正错误现场
- 在必要时用母本真源拉回主线

### 7）反向操作的性质
反向操作不属于常规同步。  
它必须视为：

- 修复动作
- 校正动作
- 回灌动作
- 救援动作

而不是默认日常动作。

### 8）反向操作前提
反向手动执行前，必须先确认：

- 回灌对象是正式资产，不是运行垃圾
- 不会把旧内容误压到当前有效现场
- 不会把错误母本反向扩散
- 必要时先做备份或快照

### 9）白名单准入
所有本地同步线都必须按明确白名单执行。  
默认不在白名单内的内容，一律不准同步。

### 10）默认不同步的内容
以下对象默认不允许进入 `System-Snapshots`：

- runtime
- `.openclaw`
- logs
- media
- cache
- 临时文件
- token / 密钥文件
- `.env.local`
- `_retired`
- 各类运行现场目录
- 任何壳层目录
- 任何非正式资产目录

## 六、新增本地同步线的准入规则

未来若新增其他需要沉淀进 `System-Snapshots` 的正式对象，允许新增对应本地同步线。

但每新增一条，都必须单独明确以下六件事：

### 1）来源是谁
必须写清来源对象到底是谁。  
不能写成含糊的“这个目录”“那边那套东西”。

### 2）目标是谁
必须写清它沉淀到 `System-Snapshots` 的哪个正式母本位。

### 3）允许进入什么
必须写清允许进入的层、路径、文件或对象范围。

### 4）禁止进入什么
必须写清禁止进入的运行垃圾、壳层、日志、媒体、缓存、密钥和临时现场。

### 5）是否保留反向手动权
原则上用户保留反向手动权，但新增同步线也应明确其适用边界。

### 6）是否进入自动链
不是所有新增同步线都默认进入自动链。  
是否自动化，必须单独判断，不得偷渡。

---

## 七、两类同步的职责边界

### 1）本地同步负责什么
本地同步负责把**运行中的正式有效资产**，干净、克制、受控地沉淀进 `System-Snapshots` 对应母本位。

### 2）GitHub 同步负责什么
GitHub 同步负责把**已经站稳在 `System-Snapshots` 内的正式母本资产**，整体版本化并留存在远端。

### 3）严禁混线
不能把 GitHub 的“整仓同步”思路，反推到本地同步。  
不能因为远端可以整仓推送，就放宽本地回写边界。  
两类同步必须永远分开理解。

---

## 八、当前明确不做

- 不做运行体整包同步
- 不做任何运行体直接推 GitHub
- 不让 `_retired`、日志、媒体、密钥进入正式同步链
- 不把临时现场目录当正式母本
- 不把自动链做成双向自动链
- 不新增未立规的新同步线
- 不用“整包简单同步”替代正式准入判断

---

## 九、一句话总规则

**GitHub 同步认的是 `System-Snapshots` 这个正式母本仓；本地同步认的是“各正式来源对象向 `System-Snapshots` 对应母本位的受控沉淀线”。当前已明确 Core brain → 02-brain 与 PG system brain → PG；未来允许扩展，但新增必须先立规、后启用。**

## 十、落地判断口径

如果以后出现混乱，先问这三个问题：

### 问题 1
这次同步，是本地同步，还是 GitHub 同步？

- 若是运行中的正式来源对象进入母本：按本地同步规则
- 若是 `System-Snapshots` 对外版本化：按 GitHub 同步规则

### 问题 2
这次同步的对象，是正式资产，还是现场残留？

- 若是正式资产：才允许进入同步判断
- 若是现场残留：直接排除

### 问题 3
这条同步线，是既有正式线，还是新增线？

- 若是既有正式线：按现有规则执行
- 若是新增线：先立规，再决定是否启用

## 十一、System-Snapshots 准入口径

### 1）准入原则
不是任何东西都可以进入 `System-Snapshots`。

只有满足以下条件的内容，才有资格被正式纳入 `System-Snapshots`：

- 已被确认属于长期保留的正式资产
- 已明确其母本身份或母本归属
- 已明确不属于运行现场残留
- 已明确其后续是否需要参与本地同步、GitHub 同步或两者之一
- 已具备相对稳定的边界，不是短期试验垃圾

### 2）不应进入的对象
以下对象原则上不应作为正式内容进入 `System-Snapshots`：

- 临时试验现场
- 一次性排障目录
- 纯运行时缓存
- 日志堆
- 媒体堆
- 密钥与环境变量文件
- 未完成收口的半成品现场
- 已确认废弃但尚未清走的旧残留

### 3）进入后的含义
一旦某项内容被正式纳入 `System-Snapshots`，就表示：

- 它已被视为正式母本体系的一部分
- 它可以进入 GitHub 同步判断范围
- 它必须接受本卡约束
- 它不能再按普通临时目录对待

## 十二、当前状态说明

### 1）当前已明确的正式本地同步线
当前已明确并在口径上成立的本地同步线为：

- `Core brain → System-Snapshots/02-brain`
- `PG system brain → System-Snapshots/PG`

### 2）当前 GitHub 同步理解
当前 GitHub 同步应理解为：

`System-Snapshots` 中所有已正式纳入、且符合准入条件的内容 → GitHub

而不是只限于某两个目录。

### 3）当前自动化状态
本卡定义的是总规则，不等于所有同步线都已经自动化落地。

是否已经自动执行、何时自动执行、由哪条脚本执行，属于后续各条同步线或各套脚本的实现问题，不由本卡自动代替。

### 4）当前收口重点
当前收口重点仍然是：

- 先把同步边界说清
- 先把正式对象与现场垃圾分清
- 先把本地同步与 GitHub 同步分清
- 在边界稳定后，再决定哪些线进入自动链
