# PG-模型切换接班规则

## 适用范围
- PG 自身模型切换
- 因监管质量、稳定性、故障而进行的切换

## 切换前
至少写一份最小交接，只保留：
- 当前模型
- 目标模型
- 当前监管主线程
- 当前停点
- 切过去后第一步确认什么

## 切换后
不能直接继续，必须先确认。

固定最小确认：
- 你现在在什么模型上？
- 你当前在接哪条监管主线程？

## 只有满足以下条件，才算接住
1. 已收到切换结果
2. 新模型能稳定回答最小确认
3. 回答链未立即进入报错、混乱、漂移

## 未接住时
必须明确说未稳定接住，不硬撑，不装成功。

## 说明
- 只继承“切换前先交接、切换后先确认、失败要明说”这条骨架
- PG 承接时还必须保留监管边界与问题判明能力

## 模型槽位正式映射与命令硬闸

### 一、正式模型槽位

- 模型0：Kimi
  - 英文模型 ID：`moonshot/kimi-k2.5`
  - 角色：PG 默认主前台 / 默认回归位 / 守门模型
  - 配置位置：`primary`

- 模型1：qwen
  - 英文模型 ID：`ollama/qwen2.5:3b`
  - 角色：本地临时辅助模型 / 快速短任务模型
  - 配置位置：`fallbacks`
  - 禁止写回 `primary`

- 模型2：GPT-5.4
  - 英文模型 ID：`openai/gpt-5.4`
  - 角色：复杂判断 / 规则复核 / 强力辅助模型
  - 配置位置：`fallbacks`
  - 禁止写回 `primary`

### 二、默认解释

用户说：

- “切换到 qwen”
- “切换到 GPT”
- “换模型1”
- “换模型2”

默认解释为：**当前会话临时热切**。

不得解释为：**永久修改默认模型**。

只有用户明确说：

- “永久修改默认模型”
- “把默认模型改成……”
- “写回 primary”

才允许讨论修改 `openclaw.json`。

### 三、命令硬闸

除非用户明确批准“永久修改默认模型”，否则禁止执行：

- `openclaw models set ...`
- `openclaw model set ...`
- `openclaw models use ...`
- `openclaw model use ...`
- 任何会改写 `openclaw.json` 或 `primary` 的命令

因为 `models set` 会把目标模型写成默认模型，不能用于临时热切。

### 四、正确处理方式

当用户要求临时切换模型时：

1. 不执行命令。
2. 提醒用户使用左上角模型下拉菜单手动切换。
3. 用户切换后，执行切换后确认：
   - 当前在什么模型上？
   - 当前接哪条监管主线程？
   - 是否出现报错、漂移、混乱？
4. 测试结束后提示是否切回模型0 Kimi。

### 五、硬结论

PG 默认主前台必须保持：

`moonshot/kimi-k2.5`

qwen 与 GPT-5.4 只能作为临时功能模型，不得因“切换”指令被写回默认模型。
